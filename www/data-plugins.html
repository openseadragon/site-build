Plugins for Data Modification

Any plugin that modifies viewer data should use tile-invalidated event, which ensures data is updated in consistent manner.

The usage is simple: you request an invalidation once you reason it is needed to update data

viewer.requestInvalidate(restoreTiles)

and provide handler for the data update

viewer.addHandler('tile-invalidated', async e => {...})

Simple Plugin Example

Example: data modification plugin

Invalidation routine is applied also when new data arrives. Therefore, we do not need to call viewer.requestInvalidate() since we do not do any updates. Simple getData() and setData() on the event object is enough here. 

const viewer = OpenSeadragon({
    id:            "example-simple",
    prefixUrl:     "/openseadragon/images/",
    navigatorSizeRatio: 0.25,
    tileSources:   "/example-images/highsmith/highsmith.dzi",
});
viewer.addHandler('tile-invalidated', async e => {
    // Get the canvas and context
    const ctx = await e.getData('context2d');
    const canvas = ctx.canvas;
    const imageData = ctx.getImageData(0, 0, canvas.width, canvas.height);
    const width = canvas.width;
    const height = canvas.height;
    
    function toGrayscale(data) {
        const grayData = new Uint8ClampedArray(data.length / 4);
        for (let i = 0; i < data.length; i += 4) {
            const avg = (data[i] + data[i + 1] + data[i + 2]) / 3;
            grayData[i / 4] = avg;
        }
        return grayData;
    }

    function sobelEdgeDetection(grayData, width, height) {
        const sobelX = [[-1, 0, 1], [-2, 0, 2], [-1, 0, 1]];
        const sobelY = [[-1, -2, -1], [ 0, 0, 0], [ 1, 2, 1]];

        const gradient = new Uint8ClampedArray(grayData.length);

        for (let y = 1; y < height - 1; y++) {
            for (let x = 1; x < width - 1; x++) {
                let gx = 0, gy = 0;

                // Apply Sobel kernels
                for (let ky = -1; ky <= 1; ky++) {
                    for (let kx = -1; kx <= 1; kx++) {
                        const pixelVal = grayData[(y + ky) * width + (x + kx)];
                        gx += pixelVal * sobelX[ky + 1][kx + 1];
                        gy += pixelVal * sobelY[ky + 1][kx + 1];
                    }
                }

                // Calculate gradient magnitude
                const magnitude = Math.sqrt(gx * gx + gy * gy);
                gradient[y * width + x] = magnitude > 255 ? 255 : magnitude;
            }
        }
        return gradient;
    }

    function drawEdges(ctx, gradient, width, height) {
        const outputImage = ctx.createImageData(width, height);
        for (let i = 0; i < gradient.length; i++) {
            const value = gradient[i];
            outputImage.data[i * 4] = value;
            outputImage.data[i * 4 + 1] = value;
            outputImage.data[i * 4 + 2] = value;
            outputImage.data[i * 4 + 3] = 255;
        }
        ctx.putImageData(outputImage, 0, 0);
    }
    
    const grayData = toGrayscale(imageData.data);
    const edges = sobelEdgeDetection(grayData, width, height);
    drawEdges(ctx, edges, width, height);
    await e.setData(ctx, "context2d");
});

Plugin Priorities

If you develop a plugin, you should make order of your event handlers configurable, so that users can control when your plugin is applied. 

const eventOrderPriority = options.eventOrderPriority;
viewer.addHandler('tile-invalidated', async e => {
    ... do something ...
}, null, eventOrderPriority);

This way, your plugin gets applied based on eventOrderPriority and thus users can control which plugin gets applied first. See handler priority argument.

Selecting the Base Data

Plugin can request, whether to start from the original data, or whether to update the current viewport state. Using viewer.requestInvalidate(false) makes your code receive the data on the viewport, so that you can stack changes. You can also request data reset at any point in the event routine using resetData(): 

viewer.addHandler('tile-invalidated', async e => {
    e.resetData(); //throw away any possible previous changes
    const freshData = await e.getData('image');
    ... etc ...
}, null, eventOrderPriority);

But beware of the data consistency, see section below.

Outdated Events

It might happen that you request invalidation while a different event still runs. You can tell simply by using: 

await viewer.requestInvalidate();  // await finishing

So if you have e.g. a range slider for user-controlled input, you might trigger too many updates at once. You can exit outdated (no longer valid) processes by checking e.outdated() flag. 

viewer.addHandler('tile-invalidated', async e => {
    ... some expensive computation ...
    if (e.outdated()) return;
    ... etc ...
}, null, eventOrderPriority);

Note though, with requestInvalidate(false) some tiles might finish before you realized the current data is no longer needed, and the viewport might show inconsistent state.

Removal of if (e.outdated()) return; will not help! The viewer also considers output of such events pointless and throws away any results. This scenario is not yet well-supported, and you must ensure no overlapping events are triggered in this specific case.

Practical considerations

With knowledge of what drawers are being used you can optimize your events. getData() might force unnecessary processing, setData() might force system to do additional type conversions. A good idea is to work with context2d type, which is flexible enough for processing, but also provides image data through canvas reference, directly usable in most drawers.

In the Depth

Understanding invalidation routine.

Update routine keeps a working cache bound to the event. This cache is not part of the system, and it is a copy of data from the tile. Once the data is ready, we atomically swap data under the renderer, and new data gets displayed. Therefore, when you finish there is one last step in the event where we ask the underlying drawer what type it requires (getRequiredDataFormats()) and potentially convert the data.

Some drawers might not be compatible with the type directly, but want to keep it nevertheless. WebGL renderer uses internal cache that keeps reference to the uploaded texture, but the cache from outside looks like a context2d or image cache (but with a weird type name). This way we keep reference to the data shown at screen and support conversion back to, for example, canvas context 2D object.

While you can add custom data to the tile using cache-related methods (get/set/create), it is probably a not good idea. Cache invalidation runs on every data-unique tile, both in the viewer and navigator. It might not be executed though on every tile.

Are all tiles updated?

No. But all data items are. Tile equality is decided using getTileCacheKey(). If this item returns equal value for two tiles, invalidation is executed only on one of the two, and references to the data are later updated on both.

So if you have same data but want to apply different processing (e.g. vignetting), you must also make sure these tiles have different (original) cache key. This will force you downloading the same data several times. However, this scenario is not expected to be frequent, and thus it is not optionally supported.

Invalidation in Drawers

If you happen to attempt implementing a drawer logics, you usually provide custom data convertors for the target type you require for rendering.

If you need the system to re-convert every data item (e.g. you upload textures as WebGL renderer and parameters have changed such as texture wrap or filter), you can also use this.viewer.requestInvalidate(). Note though, that you should also change your type name, since the system might not do anything if it realizes there is no data to update and types are fitting.

So you should change your type names and update the return value of getRequiredDataFormats(). Furthermore, you should keep for the time of being old types in getSupportedDataFormats(), and optionally remove them once processing done: 

    ...
    const newFormats = this.generateNewTypeNamesAndImplementConvertorAndDestructorLogics();
    this._supportedFormats.push(...newFormats);  // variable returned by getSupportedDataFormats
    this._requiredFormats = newFormats;          // variable returned by getRequiredDataFormats
    await this.viewer.requestInvalidate();       // forces re-execution since type has changed
    // optionally keep updated:  this._supportedFormats = newFormats;
    ...
